收银台重构 & walkman 开发流程体验心得

1. 前提

由于收银台模块划分比较明显，而业务逻辑比较繁琐，所以拿出来前端重构，改善view层代码混乱的问题

2. 模块划分

2.1 模块间包含关系
pg-pf-pay-cashier 收银台
    |-- md-pf-moguservice-float 联系客服IM挂件模块
    |-- md-pf-pay-banklist 银行列表模块
    |-- md-pf-pay-modou 胖兄的蘑豆模块
    `-- md-pf-pay-banktab 银行选项卡模块
         `-- md-pf-pay-banklist 银行列表模块
2.2 模块介绍

md-pf-pay-banklist: 传送门

md-pf-pay-banktab: 传送门

md-pf-pay-modou: 任意门

md-pf-moguservice-float: 传送门

pg-pf-pay-cashier: 凯旋门

2.3 银行列表模块（其他模块类似）

收银台中包含多种银行列表，所以银行列表作为原子模块需提供多种配置，like this

模块显示由php病量来控制，主要是为了后端友好，后端不需要关心怎么去嵌套view，只需要操作php病量即可。

// data here
$banklist1 = array(
    'banks' => array(
        // 微信
        'weipay' => array(
            ...
        ),
        // 银联
        'chinapay' => array(
            ...
        ),
        // 财付通
        'tenpay' => array(
            ...
        ),
    )
);

$banklist2 = array(
    'showlen' => 3, // 显示3个，其他全部隐藏
    'banks' => array(
        ...
    )
);

$banklist3 = array(
    'extraclass'=> 'J_departBanklist',
    'showlen' => 0, // 不显示
    'nomore' => true, // 不用默认的更多
    'banks' => array(
        ...
    )
);

这么做会增加前端工作量，把view层除业务逻辑之外的php代码都由前端来开发，但是利于代码维护，walkman里面的content.php可以直接交给后端，不用再去嵌套页面。

（实际开发中如果不是重用性非常高的模块可以不做这些，直接给后端输出好的静态文件就行，对于后端来说开发没有差别，对于前端来说静态文件版本入库，方便今后其他人来修改维护）

新开发的业务需要创建模块

walkman
    |-- common
    |-- public 开发目录，其他目录可以不用管
    `-- system

walkman后台可以创建需要定制的模块，在public安装新的包

4. 模块间依赖

依赖关系由前端自己控制，不需要交给后端N个文件，然后调了半天发现后端加载顺序错了

less

/**
 * @name pg-pf-pay-cashier
 * @desc 收银台
 * @create 2014-06-31 @buji
 */

/* 依赖关系 */
@import 'mods/md-pf-pay-banklist/index.less';
@import 'mods/md-pf-pay-banktab/index.less';
@import 'mods/md-pf-pay-modou/index.less';
@import 'mods/md-pf-moguservice-float/index.less';

.g-wrap {
    border: 1px solid #DFDFDF;
    background-color: #FFF;
    *zoom: 1;
}

...

js

/**
 * @name pg-pf-pay-cashier
 * @desc 收银台
 * @create 2014-06-31 @buji
 */     
define([
    /* 依赖关系 */
    './mods/md-pf-pay-banklist/index',
    './mods/md-pf-pay-banktab/index',
    './mods/md-pf-pay-modou/index'
], function() {
    // 收银台表单
    (function() {
        ...
    }());
});

5. 模块（页面）发布

发布页面级模块pg-pf-pay-cashier到svn，配置bo.json指定发布目录

"GruntConfig": {
    "lessPath": "./index.less",
    "jsPath": "./index.js",
    "outputCss": "public/css/pay/src/pkg-cashier.css", // svn地址
    "outputJs": "public/js/pay/src/pkg-cashier.js" // svn地址
}

然后点击压缩模块

打包后的样式脚本会自动提交到svn，pkg-开头的资源文件是由walkman打包生成的文件，文件开头会生成模块信息以及依赖关系的注释，方便找到相应模块，如：

/**
 * {
 *     modname : pg-pf-pay-cashier
 *     by : buji
 *     custom :
 *     version : 1.0.0
 *     des : 收银台
 *     dependencies : {
 *         md-pf-moguservice-float :
 *         md-pf-pay-banktab :
 *         md-pf-pay-banklist :
 *         md-pf-pay-modou :
 *     }
 * }
 */

没接触过这个项目的人知道模块名pg-pf-pay-cashier，在walkman中执行（也可以在walkman后台界面里面点安装模块）

bo install pg-pf-pay-cashier

就可获得前端最新版本以及打包配置

6. 维护

收银台是属于交易线前端项目，整个项目托管在git上，安装方法

cd $WALKMAN/public  # $WALKMAN是walkman的安装路径
curl http://gitlab.mogujie.org/f2e/docs/raw/master/_projects/install/pay/install.sh | sh

7. 优缺点 & issues

收银台的改版受到 @重楼 @阿度 的好评，但是大部分原因应该是他们没有套页面

目前svn上收银台的view目录如下：

appbeta/views/trade/cashier/
 |-- data.php
 |-- md-pf-pay-banklist.php
 |-- md-pf-pay-banktab.php
 |-- pg-pf-pay-cashier.php
 |-- md-pf-moguservice-float.php
 |-- md-pf-pay-banklist-statictpl.php
 `-- md-pf-pay-modou.php

除data.php外，其他文件都是walkman直接copy过来的，data.php是连接前后端的数据文件

data.php里 @重楼 的后端代码业务逻辑节选

/* 选中支付平台 */
if($bankId == "weipay") {
    $paytab['platform']['banklist']['banks']['weipay']['selected'] = true;
} else if($bankId == 'chinapay') {
    $paytab['platform']['banklist']['banks']['chinapay']['selected'] = true;
} else if($bankId == 'tenpay') {
    $paytab['platform']['banklist']['banks']['tenpay']['selected'] = true;
}

/* 业务逻辑，只显示微信支付 */
if($onlyShowWeixinPay == 'weixin') {
    unset($paytab['deposit']);
    unset($paytab['credit']);
    unset($paytab['platform']['banklist']['banks']['chinapay']);
    unset($paytab['platform']['banklist']['banks']['tenpay']);

    return;
}

主要操作$paytab变量，不是在dom上写业务逻辑，好处是view里面的dom干净，坏处是view的输出逻辑太依赖前端（不过前端作为最熟悉dom结构的，写这些也无可厚非）

为了更好的维护view文件，套页面由前端来写不会像现在这么乱，但是之前也重构过购物车view，后来由于后端业务逻辑的不断增加变得根本没法看了，所以这次加上了一个中间层data，除了比较重要或者比较独立的业务，其他页面最好暂时先不要写太多php，毕竟主站php框架没有模版语言，本身就不太友好，加上全部前端开发的工作量也不合适。移动端的开发以后可以尝试，避免两套模版由不同人来维护。

walkman优点：

  • 本地开发，速度快（银行列表模块是我带回家做的，没连vpn的情况下）
  • 不依赖环境，由于是本地开发，基本上属于纯静态页面，目前也支持php输出
  • 静态文件版本入库，方便后期维护，可追溯历史版本
  • 文件依赖，加载顺序由前端自己管理，不管使用了多少插件多少模块，给后端的文件最多只有pkg-xxx.js、pkg-xxx.css

walkman缺点：

  • 有一定学习成本
  • 体验还需要优化，详见issues

walkman一直内测是目前还有开发体验优化上的修改还没完成，使用流程是可以走通的，目前收银台，cps-v2已经迁移到walkman开发（青叶那边还有几个）

有什么开发需求可以添加issues传送门，也可以参与一起开发。

由于移动开发上的需求我们还不太明确，目前只增加了一套移动的模板，还没有什么项目模块，有什么好的建议也可以在issues上记下来。
